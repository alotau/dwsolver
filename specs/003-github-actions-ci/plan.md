# Implementation Plan: GitHub Actions CI — Multi-Platform Build & Test

**Branch**: `003-github-actions-ci` | **Date**: 2026-03-19 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/003-github-actions-ci/spec.md`

## Summary

DWSOLVER has no multi-platform CI. This plan implements a GitHub Actions workflow
(`.github/workflows/ci.yml`) that builds the solver and runs `tests/dw-tests.sh` on
macOS (Clang, `--enable-named-semaphores`), Linux (GCC, no flags), and Windows
(MSYS2 MinGW-w64, non-blocking) on every PR against `master` and every push to
`master`. The existing partial `build.yaml` is deleted. macOS and Linux are registered
as required status checks via branch protection rules.

## Technical Context

**Language/Version**: YAML (GitHub Actions workflow syntax)
**Primary Dependencies**: GitHub Actions; `actions/checkout@v4`; `msys2/setup-msys2@v2`
**Storage**: N/A — single workflow YAML file; no persisted state
**Testing**: Workflow itself is tested by opening a PR and observing CI run results
**Target Platform**: GitHub-hosted runners: `macos-latest`, `ubuntu-latest`, `windows-latest`
**Project Type**: CI configuration (DevOps artifact)
**Performance Goals**: macOS + Linux jobs complete within 15 minutes (SC-001)
**Constraints**: Zero secrets beyond automatic `GITHUB_TOKEN` (FR-008); Windows job non-blocking (FR-005)
**Scale/Scope**: One workflow file, three jobs, ~60 lines of YAML

## Phase 0: Research Findings

All unknowns resolved. See [research.md](research.md) for full detail.

Key decisions:
- **R1**: Default branch is `master` (not `main`); spec.md updated accordingly
- **R2**: `configure` is pre-generated — no autotools install step needed in CI
- **R3**: Delete `build.yaml`; `ci.yml` supersedes it
- **R4**: Invoke tests as `bash tests/dw-tests.sh` (not `./tests/dw-tests.sh`) — Ubuntu's `sh` is dash; `pushd`/`popd` require bash
- **R5**: Add `src/` to `$GITHUB_PATH` after `make`; avoids `sudo make install`
- **R6**: macOS and Linux runners have toolchains pre-installed; zero setup steps needed
- **R7**: Windows: `msys2/setup-msys2@v2` with `MINGW64`, packages `mingw-w64-x86_64-gcc make`, `cache: true`
- **R8**: Pin `actions/checkout@v4`, `msys2/setup-msys2@v2`

### Notes on Windows Job

The Windows job is expected to fail initially because:
1. KD-001 (multiple-definition in `dw.h`) causes link errors on all non-macOS platforms — fixed by spec 002
2. There may be additional Windows-specific platform issues addressed by spec 004

The `continue-on-error: true` flag ensures Windows failures do not block PRs.
Once spec 002 and spec 004 land, the flag can be removed and Windows can be promoted
to a required check.
