# Implementation Plan: OpenSSF Best Practices Badge (Passing Level)

**Branch**: `015-openssf-passing-badge` | **Date**: 2026-03-24 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `/specs/015-openssf-passing-badge/spec.md`

## Summary

Achieve the OpenSSF Best Practices Badge (Passing level) for the `dwsolver` / `libdwsolver` project. The project already satisfies most Passing criteria (GPL-3.0 OSI license, public GitHub repository, CI matrix across macOS/Linux/Windows, Docker build, test suite, static analysis with clang-tidy + cppcheck, AddressSanitizer + TSan CI jobs, distributed VCS). The remaining work falls into five concrete buckets: (1) create `SECURITY.md` and `CONTRIBUTING.md`, (2) add SPDX license identifiers to all 17 source files, (3) establish a structured release-notes format with a mandatory Security subsection, (4) register the project on the OpenSSF badge platform and complete the self-assessment, and (5) update the README with the Passing badge widget. No solver logic is touched; this is entirely documentation, CI evidence, and project metadata.

**Structure Decision**: Single-project layout; no new directories in `src/` or `tests/`. New documentation files go to the repository root (matching GNU project conventions already in use for `AUTHORS`, `COPYING`, `NEWS`, `ChangeLog`). SPDX identifiers are appended as the last line of each file's existing `/* ... */` license header block, after the closing `***/` line — keeping the existing comment block intact and simply adding one machine-readable line.

**Note on ASAN CI (FR-007)**: The `linux-asan-ubsan` job already exists in `ci-linux.yml` and runs `dw-tests.sh` under `-fsanitize=address,undefined`. This satisfies the `dynamic_analysis` and `dynamic_analysis_unsafe` Passing criteria *as-is* — the only work is cross-referencing it in the OpenSSF self-assessment (FR-008). No CI changes are required.
